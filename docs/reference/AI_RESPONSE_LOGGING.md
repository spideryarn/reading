# AI Response Logging

This document describes the canonical approach for capturing and storing **all** AI responses generated by Spideryarn Reading.

## Overview
Every LLM invocation is represented by a row in the `ai_calls` table.  The row stores:

1. `raw_api_response` (JSONB) – the **entire** unmodified response object returned by Vercel AI SDK, including provider-specific metadata.
2. `latency_ms` – execution latency in milliseconds.
3. `prompt_tokens`, `completion_tokens`, `total_tokens`, `reasoning_tokens` – token usage metrics.
4. Other bookkeeping fields (`prompt_type`, `model_string`, etc.).

All writes to `ai_calls` must go through **`AIResponseLogger.completeAICall()`**.  Direct calls to `AiCallService.completeCall()` are forbidden (ESLint rule pending).

## Latency Calculation
Priority order:

1. `finishTimestamp - startTimestamp` (if provided by the SDK).
2. `experimental_providerMetadata.latency` (provider-supplied value).
3. **Manual fallback**: we record `manualStartTimestamp` and `manualFinishTimestamp` **around each `generateText()` invocation** in `lib/prompts/types.ts`.  These timestamps are injected into `raw_api_response` when the SDK omits them.

> Why not rely solely on the SDK?  Field testing in production showed the SDK currently omits `startTimestamp`/`finishTimestamp` in all responses.  Rolling our own guarantees consistent latency tracking today while remaining compatible with future SDK updates.

## Query Examples

```sql
-- Hourly latency distribution for the last 24 h
SELECT
  date_trunc('hour', created_at)  AS hr,
  percentile_cont(0.50) WITHIN GROUP (ORDER BY latency_ms) AS p50_ms,
  percentile_cont(0.95) WITHIN GROUP (ORDER BY latency_ms) AS p95_ms,
  COUNT(*)                                   AS calls
FROM ai_calls
WHERE created_at > now() - interval '24 hours'
GROUP BY hr
ORDER BY hr DESC;
```

## Debugging Tips
* Use `scripts/verify-ai-response-logging.ts` to sanity-check recent rows.
* Inspect individual rows:
  ```sql
  select id, raw_api_response->>'finishReason', latency_ms
  from ai_calls
  order by created_at desc limit 5;
  ```
